# План тестирования системы "Портфолио исследователя"

## 1. Введение

### 1.1 Цель документа
Данный документ описывает стратегию, подходы, объем и расписание тестирования системы управления портфолио исследователя. План предназначен для обеспечения качества программного продукта и соответствия требованиям ТЗ.

### 1.2 Область тестирования
Тестированию подлежат все модули системы:
- Модуль управления записями портфолио
- Система ключевых слов
- Исследовательская карта
- Система достижений
- Трекер компетенций
- Дашборд компетенций
- Экспорт отчетов
- Управление целями

### 1.3 Критерии начала тестирования
- [ ] Завершена разработка всех модулей
- [ ] База данных настроена и заполнена тестовыми данными
- [ ] Документация по установке готова
- [ ] Определены тестовые сценарии

### 1.4 Критерии завершения тестирования
- [ ] Выполнены все запланированные тест-кейсы
- [ ] Уровень успешных тестов ≥ 95%
- [ ] Критические и блокирующие дефекты устранены
- [ ] Подготовлен отчет о тестировании

## 2. Стратегия тестирования

### 2.1 Типы тестирования

#### 2.1.1 Функциональное тестирование
- Проверка соответствия функциональным требованиям
- Тестирование позитивных и негативных сценариев
- Проверка граничных значений

#### 2.1.2 Интеграционное тестирование
- Тестирование взаимодействия с PostgreSQL
- Проверка интеграции модулей между собой
- Тестирование экспорта в Word

#### 2.1.3 Системное тестирование
- Тестирование системы в целом
- Проверка выполнения бизнес-сценариев
- Тестирование производительности

#### 2.1.4 Приемочное тестирование
- Проверка соответствия требованиям заказчика
- Тестирование ключевых сценариев использования

#### 2.1.5 Регрессионное тестирование
- После исправления дефектов
- При добавлении нового функционала

### 2.2 Подходы к тестированию
- **Ручное тестирование:** Для GUI и сложных сценариев
- **Автоматизированное тестирование:** Для регрессионных тестов (если будет реализовано)
- **Исследовательское тестирование:** Для выявления неочевидных дефектов

### 2.3 Риски и их минимизация

| Риск | Вероятность | Влияние | Меры минимизации |
|------|------------|---------|------------------|
| Неполные требования | Средняя | Высокое | Тесное взаимодействие с заказчиком |
| Проблемы с БД | Низкая | Высокое | Резервные копии, тестовая БД |
| Производительность | Средняя | Среднее | Раннее тестирование производительности |
| Совместимость ОС | Низкая | Низкое | Тестирование на разных ОС |
| Ошибки экспорта | Средняя | Высокое | Детальное тестирование экспорта |

## 3. Тестовая среда

### 3.1 Аппаратное обеспечение
- **ОС:** Windows 10/11, Ubuntu 20.04+
- **Процессор:** Intel Core i5 или эквивалент
- **Память:** 8 ГБ RAM
- **Хранилище:** 10 ГБ свободного места

### 3.2 Программное обеспечение
- **Python:** 3.8+
- **PostgreSQL:** 12+
- **pgAdmin 4:** Последняя стабильная версия
- **LibreOffice/Word:** Для проверки экспортируемых файлов

### 3.3 Тестовые данные
- **Тестовая БД:** Отдельная база данных `21ис_test`
- **Набор данных:** 50+ тестовых записей различных типов
- **Пользователи:** 3 тестовых пользователя с разными наборами данных

## 5. Расписание тестирования

### 5.1 Фазы тестирования

| Фаза | Сроки | Ответственный | Результат |
|------|-------|---------------|-----------|
| Подготовка тестовой среды | День 1-2 | QA инженер | Готовая тестовая среда |
| Модульное тестирование | День 3-5 | Разработчик | Отчет о модульном тестировании |
| Интеграционное тестирование | День 6-7 | QA инженер | Отчет об интеграционном тестировании |

### 5.2 Ресурсы
- **QA инженер:** 1 человек
- **Разработчик:** 1 человек (для исправления дефектов)
- **Тестеры:** 2 человека (включая заказчика)
- **Оборудование:** 2 тестовые машины

## 6. Отчетность

### 6.1 Документы для предоставления
1. **Ежедневный отчет:** Прогресс тестирования, найденные дефекты
2. **Отчет о тестировании:** По завершении каждой фазы
3. **Итоговый отчет:** По завершении всех тестов
4. **Матрица покрытия требований:** Соответствие тестов требованиям

### 6.2 Отслеживание дефектов
**Инструмент:** Таблица Excel / Google Sheets
**Поля для дефектов:**
- ID дефекта
- Название
- Описание
- Шаги воспроизведения
- Ожидаемый результат
- Фактический результат
- Серьезность (Критический, Высокий, Средний, Низкий)
- Приоритет (Высокий, Средний, Низкий)
- Статус (Открыт, В работе, Исправлен, Проверен, Закрыт)
- Модуль
- Дата обнаружения
- Ответственный

### 6.3 Критерии серьезности дефектов
- **Критический:** Система не запускается, потеря данных, блокирующая ошибка
- **Высокий:** Основная функция не работает, некорректные расчеты
- **Средний:** Функция работает с ограничениями, проблемы с UI
- **Низкий:** Косметические проблемы, опечатки

## 7. Критерии приемки

### 7.1 Технические критерии
- [ ] Все функциональные требования реализованы
- [ ] Успешное выполнение ≥95% тест-кейсов
- [ ] Отсутствие критических и блокирующих дефектов
- [ ] Все высокоприоритетные дефекты исправлены
- [ ] Система работает стабильно минимум 8 часов без сбоев

### 7.2 Бизнес-критерии
- [ ] Подтверждение заказчиком ключевых сценариев
- [ ] Соответствие требованиям производительности
- [ ] Корректность экспортируемых отчетов
- [ ] Удовлетворительная оценка удобства использования

## 8. Риски и ограничения

### 8.1 Ограничения тестирования
- Невозможность тестирования на всех версиях ОС
- Ограниченные ресурсы для нагрузочного тестирования
- Отсутствие автоматизированных тестов для GUI

### 8.2 План действий при рисках
1. **Нехватка времени:** Сосредоточиться на критическом функционале
2. **Проблемы с БД:** Использовать резервные копии
3. **Неполные требования:** Уточнять у заказчика в процессе тестирования
4. **Сложные дефекты:** Привлечь разработчика для совместного анализа
